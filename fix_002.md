## 1. Duplicate `get_env()` Implementation (Priority: High)

### Issue
`scripts/env_loader.py` provides `get_env()` but `scripts/create_bot.py` and `backtest/examples/yfinance_backtest.py` each import `scripts.env_loader` separately when they could use a single shared import.

### Current State
- `scripts/env_loader.py` has `get_env()` function
- Multiple scripts re-import the same module
- Inconsistent patterns across files

### Recommendation
Create a single `scripts/__init__.py` that exports `get_env`:
```python
# scripts/__init__.py
from .env_loader import get_env, get_required_env
```

This would allow:
```python
# Instead of: from scripts.env_loader import get_env
# Use:
from scripts import get_env
```

### Files to Update
- `scripts/__init__.py` (create)
- `scripts/create_bot.py`
- `backtest/examples/yfinance_backtest.py`
- `backtest/examples/polygon_backtest.py`
- `bots/nq_open_gate_001/main.py`

## 4. `backtest/backtest.py` - Unused Return Value in `walk_forward()` (Priority: Medium)

### Issue
The `walk_forward()` method returns a dict with train/test results, but the actual data is already accessible through `self.result`. The return structure duplicates information.

### Current Code
```python
def walk_forward(self, data: pd.DataFrame, train_pct: float = 0.7):
    train_size = int(len(data) * train_pct)
    train = data.iloc[:train_size]
    test = data.iloc[train_size:]

    train_result = self.run(train)
    test_result = self.run(test)

    return {
        "train": train_result,
        "test": test_result,
        "train_size": len(train),
        "test_size": len(test),
    }
```

### Recommendation
Simplify to return only the results (train/test) without extra metadata:
```python
def walk_forward(self, data: pd.DataFrame, train_pct: float = 0.7):
    train_size = int(len(data) * train_pct)
    train = data.iloc[:train_size]
    test = data.iloc[train_size:]

    return {"train": self.run(train), "test": self.run(test)}
```

- `backtest/backtest.py`

## 7. `scripts/create_bot.py` - Hardcoded Default for `num` Extraction (Priority: Low)

### Issue
The `create_bot()` function extracts the bot number using `bot_id.split("-")[-1]` with a fallback to `"002"`, which is fragile if bot ID format changes.

### Current Code
```python
num = bot_id.split("-")[-1] if bot_id else "002"
```

### Recommendation
Add validation or use a more robust regex:
```python
import re
# Extract trailing digits
match = re.search(r"(\d+)$", bot_id)
num = match.group(1) if match else "002"
```

- `scripts/create_bot.py`

## Supplemental Review - Architectural & Performance Deep Dive

**Date:** 2026-04-06  
**Focus:** Scalability, Async Performance, and Strategy Integrity

Following the initial review, a deeper analysis of the core execution and strategy layers reveals several critical bottlenecks and architectural improvements required for production readiness.

### 15. Async Blocking in Alerting (Priority: Critical)

**Issue:** `shared/core/alerts.py` uses the synchronous `requests` library.
**Impact:** A slow Discord/Slack webhook response (e.g., 5-10s timeout) will block the entire `asyncio` trading loop, preventing the bot from processing market data or exiting positions during that window.
**Recommendation:** Replace `requests` with `httpx` or `aiohttp` to ensure non-blocking notifications.

### 16. Subprocess Overhead in Env Loading (Priority: High)

**Issue:** `scripts/env_loader.py` calls `subprocess.run(["varlock", ...])` for every individual variable request via `get_env`.
**Impact:** Significant latency at startup and during runtime if variables are checked frequently.
**Recommendation:** Load all required environment variables into a cache at startup using `varlock load` once, then serve subsequent requests from memory.

### 17. Database Connection Management (Priority: Medium)

**Issue:** `shared/core/state.py` opens and closes a new SQLite connection for every `get()`, `put()`, and `record_trade()` call.
**Impact:** Disk I/O overhead and latency.
**Recommendation:** Maintain a single persistent connection per bot instance or use `aiosqlite` for non-blocking database operations.

### 18. Strategy Confirmation Logic "Ghost" Code (Priority: Medium)

**Issue:** `OpenGateStrategy.check_confirmation()` (handling wick rejections/engulfing) is implemented but never called in the main `on_data()` loop.
**Impact:** The strategy enters on a simple breakout/retest without the intended confirmation patterns, leading to potential "fake-out" entries.
**Recommendation:** Integrate `check_confirmation` into the `on_data` flow before triggering an entry signal.

### 19. Boilerplate in Bot Entry Points (Priority: Medium)

**Issue:** `bots/nq_open_gate_001/main.py` contains ~100 lines of setup and loop logic that will be identical for other bots.
**Impact:** High maintenance cost and bug propagation across multiple bots.
**Recommendation:** Create a `shared.core.bot.BotRunner` class to handle the lifecycle, heartbeat, and signal-to-execution mapping.

### 20. Vectorbt vs. Strategy Drift (Priority: Low)

**Issue:** `backtest/vectorbt_optimizer.py` uses a hardcoded SMA crossover instead of the actual `OpenGate` strategy logic.
**Impact:** Optimization results from `vectorbt` are irrelevant to the actual bot performance.
**Recommendation:** Vectorize the `OpenGate` logic (using NumPy/Pandas) so `vectorbt` can sweep the actual strategy parameters.
