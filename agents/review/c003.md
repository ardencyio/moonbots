## Overall Verdict: REQUEST CHANGES (one showstopper remains)

The file-level work on the HMM stack has visibly improved since c001. The headline wins are real: `bfill()` is gone from the feature pipeline, the rolling z-score is causal with `shift(1)`, the label sort is stable via `np.lexsort`, strategy allocations/leverage now match the spec matrix, `n_init` vs `n_iter` is no longer conflated, the orchestrator cache is populated on first use, hyper-params round-trip through `save`/`load`, and inference state is thread-local.

**However:** the core of the system — the forward-algorithm transition step — is still mathematically wrong in direction. It computes `Σᵢ α[i]·T[j,i]` instead of `Σᵢ α[i]·T[i,j]`. Every posterior after bar 0 is derived from the transpose of the learned transition matrix. The existing no-look-ahead test cannot catch it (both halves of the comparison use the same wrong formula and therefore agree on the wrong answer). This must be fixed and guarded by a regression test before the engine is trusted for any backtest decision.

Integration per MULTI.md is also still on paper: `bots/hmm_spy_001/` is empty, `BotProcess` is hard-wired to the NQ bot's module path, and there is no Alpaca execution handler, no live market-data shim, and no `hmm_regime_strategy.py` adapter for the existing `BacktestRunner`.

## Critical Findings

### C-1 — Forward algorithm applies the transition matrix in the wrong direction
`shared/core/hmm/hmm_engine.py:479-482`

```python
log_transmat = np.log(model.transmat_.T + 1e-300)
log_alpha_transitioned = logsumexp(
    log_alpha_prev[:, np.newaxis] + log_transmat, axis=0
)
```

`hmmlearn` stores `transmat_[i, j] = P(state_t = j | state_{t-1} = i)`. The forward step requires
`α_t[j] = Σᵢ α_{t-1}[i] · T[i, j]`.

Trace the broadcast: `log_alpha_prev[:, None]` has shape `(n, 1)`; `log_transmat = log(Tᵀ)` has shape `(n, n)` where `Tᵀ[i, j] = T[j, i]`. Their sum at position `[i, j]` is
`log α_{t-1}[i] + log T[j, i]`. `logsumexp(..., axis=0)` sums over `i`, producing
`log Σᵢ α_{t-1}[i] · T[j, i]` — i.e. `(T @ α)[j]`, not `(α @ T)[j]`.

For symmetric `T` the two match; for the sticky-with-fast-exit regime topology this bot is designed to detect, they disagree. The inline comment in the code ("log_transmat.T[j,i] = log P(state_t=j | state_{t-1}=i)") is itself wrong — `log_transmat.T` is `log(T)`, whose element `[j, i]` is `log P(state_t=i | state_{t-1}=j)`, the opposite mapping. The engine will silently return biased posteriors; `TestNoLookAheadBias.test_no_lookahead_regime_identity` passes because both runs apply the same wrong step.

**Fix**
```python
log_trans = np.log(model.transmat_ + 1e-300)      # T[i,j] = P(i→j)
log_alpha_transitioned = logsumexp(
    log_alpha_prev[:, np.newaxis] + log_trans,    # [i,j] = log α[i] + log T[i,j]
    axis=0,
)                                                 # sum over i
```

**Regression test** (must be added):

```python
# Seed HMMEngine with a known asymmetric transmat (e.g. sticky BULL, fast→CRASH)
# and compare predict_filtered at bar T to an independent reference implementation
# (hmmlearn's own predict_proba run on data[0:T+1]).
```

## High-Severity Findings

### H-1 — `Signal` declared but never emitted
`regime_strategies.py:33-56` defines the spec-shaped `Signal` dataclass with `direction`, `confidence`, `position_size_pct`, `leverage`, `regime_id`, `regime_name`, `regime_probability`, `timestamp`, `reasoning`, `strategy_name`, `metadata`. `BaseRegimeStrategy.generate_signal` at `regime_strategies.py:74` still returns `StrategySignal`. Downstream code that is supposed to consume `Signal` per spec cannot read `leverage`, `regime_id`, etc. as structured fields. Either delete `Signal` or migrate strategies to produce it.

### H-2 — Uncertainty mode halves size but does not force leverage to 1.0x
`regime_strategies.py:495-501`

```python
is_uncertain = confidence < min_confidence or is_flickering
if is_uncertain:
    signal.position_size *= 0.5
    signal.reason = f"{signal.reason} [UNCERTAINTY — size halved]"
```

Spec (Phase 3 §"Confidence and Uncertainty") requires *both* halve the size and force leverage → 1.0x. Because `StrategySignal` has no `leverage` field at all, the leverage override is not even reachable in the current API. This interacts with H-1: fixing by migrating to `Signal` also unlocks the correct leverage override.

### H-3 — `LABEL_TO_STRATEGY` dict is absent
Spec Phase 3 §"Keep backward-compatible aliases" lists individual class aliases *and* a `LABEL_TO_STRATEGY` dict keyed by every possible label. Aliases exist (`regime_strategies.py:351-355`); the dict is not defined. Any caller that resolves strategy by label name will fail.

### H-4 — ADX ±DM computation is mutually contaminated
`feature_engineering.py:136-145`

```python
plus_dm  = high - high.shift(1)             # raw
minus_dm = low.shift(1) - low               # raw
plus_dm  = pd.Series(np.where((plus_dm > minus_dm) & (plus_dm > 0), plus_dm, 0), ...)   # mutates plus_dm
minus_dm = pd.Series(np.where((minus_dm > plus_dm) & (minus_dm > 0), minus_dm, 0), ...)  # reads MUTATED plus_dm
```

On any bar where raw `minus_dm` beat raw `plus_dm`, `plus_dm` is first zeroed. The second line's `minus_dm > plus_dm` then becomes `minus_dm > 0`, a weaker condition, so *both* ±DM can be non-zero on the same bar — violating the ADX definition. Compute the masked arrays from the original raw values before either is written back.

### H-5 — `reset_inference_state()` is not called at the end of `train()`
`hmm_engine.py:174-273`

After a retrain the in-memory `_previous_proba` still reflects the *old* model's state space. The next `predict_filtered` call will use that posterior as the prior for the *new* model's forward step. When `n_components` differs between models, `log_alpha_prev[:, None] + log_trans` broadcasts across mismatched shapes and produces silently wrong output. Add `self.reset_inference_state()` to the tail of `train()` and log it at INFO.

### H-6 — `scipy.linalg.LinAlgError` not caught on singular covariance
`hmm_engine.py:428-437`, `hmm_engine.py:438-444`

The `full` branch wraps `scipy.linalg.solve` with `except np.linalg.LinAlgError`, but scipy raises `scipy.linalg.LinAlgError`, which is *not* a subclass of `np.linalg.LinAlgError` in scipy ≥ 1.9. A singular covariance will crash `predict_filtered` rather than downgrading that state's emission to `-inf`.

The `tied` branch has *no* `try/except` at all (`hmm_engine.py:438-444`). Also: `np.linalg.det(cov)` underflows to 0 in high feature dimensions, turning `log(det)` into `-inf` silently. Use `np.linalg.slogdet`, and broaden the except to `(np.linalg.LinAlgError, ValueError, ScipyLinAlgError)`.

### H-7 — `predict_filtered` has no input validation
Three silent-failure modes:

1. **NaN features** flow through `solve` → NaN Mahalanobis → NaN emission → `argmax(NaN)` returns 0. Engine happily reports regime 0 with `probability=NaN`.
2. **Wrong feature count** hits `model.means_[i]` broadcast mismatch, raising `ValueError`, which is not caught.
3. **Stale `_previous_proba` after a retrain with different `n_components`** — see H-5.

Validate shape and finiteness at the top of `predict_filtered`.

### H-8 — BIC accepts non-converged models
`hmm_engine.py:212-236`

The candidate loop never checks `hmm.monitor_.converged`. When *all* inits for a given `n_components` fail to converge, the "best-of-inits" model is chosen on partial log-likelihood — a poor proxy for fit. Log convergence status and prefer converged candidates when at least one exists.

### H-9 — `test_look_ahead.py` absent as a standalone file
The HMM spec's "MANDATORY TEST" section explicitly names the file `tests/test_look_ahead.py` (or in MULTI.md, `tests/hmm/test_look_ahead.py`). The test class `TestNoLookAheadBias` lives inside `tests/hmm/test_hmm_engine.py`. Beyond the naming gap, the existing test slices the *already-normalised* features DataFrame, so it cannot detect any look-ahead that happens *upstream* of `predict_filtered` (for example, a future regression of `normalize_features`). The spec's intent is to slice *OHLCV*, recompute features on each slice, and verify regime identity at `T`. Add a file-level `tests/hmm/test_look_ahead.py` that does that.

### H-10 — `test_hmm_strategy.py` absent
Spec requires coverage for the strategy layer: orchestrator vol-rank mapping, uncertainty mode size halving + leverage clamp, rebalancing threshold, backward-compat aliases, `update_regime_infos` re-mapping after retrain. None of this is tested.

## Medium Findings

- **`rsi_zscore` is z-scored twice.** `feature_engineering.py:61-63` computes a rolling z-score of RSI into the column, then `normalize_features` at `:176-179` applies a second rolling z-score to the same column. The second normalisation dampens the designed signal.
- **`needs_rebalance` churns when flat.** `regime_strategies.py:515-521` returns `True` whenever `last_position_size == 0.0`, including the zero-to-zero case. After an exit, every subsequent "target=0" call triggers a no-op rebalance path. Guard with `if target == 0.0 and last == 0.0: return False`.
- **`min_train_bars * 2` silently doubles the caller's intent.** `hmm_engine.py:187`. A caller passing `min_train_bars=504` gets a 1008-bar requirement. The docstring at `hmm_engine.py:82` warns about this, but it is still a foot-gun. Either drop the `* 2` or rename the parameter.
- **`RegimeState.timestamp = datetime.now()`** (`hmm_engine.py:562`) bakes wall-clock time into backtests. Accept an optional `timestamp` argument and thread the bar time through.
- **`save`/`load` persists a subset of `RegimeInfo`.** Lines 609-617 serialise only `regime_id`, `regime_name`, `expected_return`, `expected_volatility`; `recommended_strategy_type`, `max_leverage_allowed`, `max_position_size_pct`, `min_confidence_to_act` are dropped.
- **Numerical stability guard is reactive, not preventive.** `hmm_engine.py:494-500` catches non-finite `log_normalizer` after the fact. Over long sequences with peaked posteriors, `log(0 + 1e-300) ≈ -690`; with 5-state `full` covariance on high-dimensional features you can approach `float64 exp` underflow every bar. Consider also clipping `log_alpha_unnorm` before `logsumexp`.
- **`get_regime_flicker_rate()` returns a count, not a rate.** `hmm_engine.py:568-577`. Matches the `flicker_threshold` (an int count in settings) but the name implies a normalised rate.
- **Dead computation.** `high_low_range` (`feature_engineering.py:38`) is computed but not listed in `hmm_features`. Remove or include.
- **`scipy.linalg.solve` imported inside the per-state loop.** `hmm_engine.py:428, 440`. Move to module top.
- **`min_confidence` is carried on both `HMMEngine` and `RegimeState`.** They can diverge if the caller constructs a `RegimeState` manually; document the source of truth.

## Positive Observations

- **Forward-algorithm *form*** — log-space, cached prior, `logsumexp` normalisation, explicit per-covariance-type emission calculation — is well-structured. Only the transition direction (C-1) is wrong; the surrounding plumbing is sound.
- **Look-ahead hygiene upstream of the forward pass** is now correct: no `bfill`, causal rolling z-score with `shift(1)`, module docstring warning against `ffill/bfill` synthetic warm-up data.
- **Label vs. strategy orthogonality** is preserved: `_label_regimes` sorts by return, `StrategyOrchestrator.update_regime_infos` sorts by volatility independently.
- **Stable regime labelling** via `np.lexsort((arange, returns))` — tie-broken deterministically on regime id.
- **Thread-local inference state** (`_thread_local`) is the right design: trained model shared, per-caller inference state isolated, `reset_inference_state` exposed as a boundary method.
- **Strategy matrix matches the spec** (allocations 0.95 / 0.95-or-0.60 / 0.60, leverage 1.25 / 1.0 / 1.0, ATR-and-EMA-anchored stops).
- **Orchestrator cache** populates on first call (`cache_empty or cache_stale`) and `int(regime_info.regime_id)` dodges the np.int64 key-mismatch class of bug.
- **BIC parameter counts** are correct per covariance type.
- **`save`/`load` now persists hyper-parameters**, so a model trained with non-default `min_confidence` will not silently revert on reload.
- **Backward-compat aliases are in place** (individual class aliases), keeping label-keyed callers working.
