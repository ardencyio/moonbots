## Overall Verdict: REQUEST CHANGES

This is a structurally-sound first pass. The forward algorithm — the most critical piece of the spec — is implemented correctly. The regime-labeling-by-return vs strategy-selection-by-volatility separation (the key design insight from MULTI.md) is also done right. However, there are two CRITICAL look-ahead bias leaks in the feature pipeline that undermine the otherwise-correct HMM, and the strategy layer is materially misaligned with the spec's allocation/leverage matrix.

## Look-Ahead Bias Assessment

**There IS look-ahead bias in the pipeline.** Not in the forward algorithm itself (correctly implemented) — but two upstream leaks in feature preprocessing neutralize it:

### CRITICAL-1 — `bfill()` in `compute_features`
`feature_engineering.py:81`
```python
df = df.bfill().ffill().fillna(0)
```
`bfill()` fills NaN values with **future** bar values. Every early rolling window (the first 199+ bars, due to `sma_200`) is overwritten with values from later bars. This is a textbook look-ahead source.

**Fix:** Remove `bfill()`. Use only `ffill()` or `.dropna()` to skip the warm-up window.

### CRITICAL-2 — Non-rolling (global) z-score normalization
`feature_engineering.py:136–150`
The spec requires 252-period **rolling** z-scores. The implementation uses `sklearn.StandardScaler.fit_transform(features)` over the entire DataFrame at once — mean and std are computed including future bars.

Compounding this: the scaler is never persisted in `HMMEngine.save()` / `.load()`, so live inference cannot reproduce training normalization.

**Fix:** Causal rolling z-score:
```python
mean = features.rolling(252, min_periods=60).mean().shift(1)
std  = features.rolling(252, min_periods=60).std().shift(1)
normalized = (features - mean) / std
```

### Consequence for the look-ahead test
`TestNoLookAheadBias.test_no_lookahead_regime_identity` (test_hmm_engine.py) currently passes not because the pipeline is unbiased, but because both short and long slices run through the same non-causal preprocessing separately. The true test must slice **OHLCV data** (not pre-computed features) and recompute features from scratch on each slice.

## Critical Findings

### CRITICAL-3 — `RegimeState.is_uncertain` crashes at runtime
`hmm_engine.py:65–66`
```python
@property
def is_uncertain(self) -> bool:
    return self.probability < self.__class__.__annotations__["probability"].__doc__
```
This compares a `float` against `float.__doc__` (a string). Any caller of `is_uncertain` will raise `TypeError`. Intended comparison target is the `min_confidence` threshold (0.55 from spec).

## High-Severity Findings

### HIGH-1 — Strategy allocations and leverage violate spec
`regime_strategies.py` (all three strategy classes)

The spec (Phase 3) defines a precise allocation/leverage matrix:

| Strategy | Allocation | Leverage |
|---|---|---|
| LowVolBullStrategy | 0.95 | 1.25x |
| MidVolCautiousStrategy | 0.95 if price > 50 EMA, else 0.60 | 1.0x |
| HighVolDefensiveStrategy | 0.60 (stays LONG) | 1.0x |

What is implemented:
- `LowVolBullStrategy`: `default_leverage=2.0`, `max_position_pct=0.30` — over-levered
- `MidVolCautiousStrategy`: `default_leverage=1.2` — wrong leverage, no trend/EMA check
- `HighVolDefensiveStrategy`: `default_leverage=0.7`, `max_position_pct=0.10` — spec explicitly says "stay 60% invested to catch V-shaped rebounds"; this shrinks to 10%

Also: stops use `volatility * multiplier`. Spec requires ATR-based stops anchored to the 50 EMA (`50 EMA - 0.5 ATR` for low vol, `50 EMA - 1.0 ATR` for high vol).

### HIGH-2 — Wrong `Signal` dataclass
`regime_strategies.py:16–25` defines `StrategySignal` with fields `{action, position_size, stop_loss, take_profit, reason}`. The spec's `Signal` dataclass requires:
```
symbol, direction, confidence, entry_price, stop_loss, take_profit,
position_size_pct, leverage, regime_id, regime_name, regime_probability,
timestamp, reasoning, strategy_name, metadata
```
Downstream risk/execution code expecting the spec shape will not interop.

### HIGH-3 — No confidence/uncertainty mode
Spec Phase 3 §"Confidence and Uncertainty":
- When `prob < 0.55` OR `is_flickering`: halve all position sizes, force leverage to 1.0x, append `"[UNCERTAINTY — size halved]"` to reasoning.

The `uncertainty_strategy` field exists but no logic checks probability thresholds or flickering state. `generate_signal` doesn't even receive `regime_state` or `is_flickering`.

### HIGH-4 — No rebalancing threshold
Spec: only rebalance when `|target_allocation - current_allocation| > 10%`. No such logic exists in `StrategyOrchestrator`. Every bar that nudges a probability will churn unnecessary trades.

### HIGH-5 — Missing `update_regime_infos` on orchestrator
Spec requires `StrategyOrchestrator.update_regime_infos(regime_infos)` to rebuild the vol-rank map after HMM weekly retraining (Phase 7, main loop step 11). Without it, the strategy mapping silently stays stale after every retrain — applying old regime strategies to new regime IDs. In live trading this could be catastrophic.

### HIGH-6 — `np.int64` vs `int` key mismatch in `regime_infos`
`hmm_engine.py:307` stores regime IDs as `np.int64` (from `np.argsort`). `StrategyOrchestrator.get_strategy_for_regime` does `if regime_id not in regime_infos` — a plain `int` will not match a `np.int64` key, so the lookup falls through to `uncertainty_strategy` silently every time.
Fix: `self.regime_infos[int(regime_id)] = ...`

### HIGH-7 — Backward-compat aliases missing
Spec requires:
```python
CrashDefensiveStrategy = HighVolDefensiveStrategy
BearTrendStrategy = HighVolDefensiveStrategy
MeanReversionStrategy = MidVolCautiousStrategy
BullTrendStrategy = LowVolBullStrategy
EuphoriaCautiousStrategy = LowVolBullStrategy
LABEL_TO_STRATEGY = {...}
```
None of these are defined anywhere.

### HIGH-8 — Feature set misaligned with spec

| Spec feature | Status |
|---|---|
| log returns 1, 5, 20 | Only `return_1` included |
| vol ratio (5-period / 20-period) | Uses 20/60 window instead |
| volume z-score vs 50-period mean | Not included (uses `volume / volume_ma20`) |
| volume trend (10-SMA slope) | Missing |
| 50-SMA slope | Not implemented (has SMA level comparison) |
| RSI(14) z-score | Raw RSI used, not z-scored |
| distance from 200 SMA as % of price | Computed but not in `hmm_features` |
| ROC 10, ROC 20 | Computed but not selected |
| `up_days_ratio`, `price_vs_sma20` | Present but not in spec |

### HIGH-9 — RSI and ADX divide-by-zero
`feature_engineering.py:52–55`: `rs = gain / loss` — when all bars in the lookback are up, `loss == 0`, producing `inf`. Same issue at `:130` for ADX: `dx = 100 * abs(plus_di - minus_di) / (plus_di + minus_di)` where both are zero on flat bars.

### HIGH-10 — Missing test files
- `tests/hmm/test_look_ahead.py` — **spec-mandated** as the "MANDATORY TEST"; does not exist
- `tests/hmm/test_hmm_strategy.py` — required by spec; does not exist

## Medium Findings

**`n_iter` vs `n_init` confusion** (`hmm_engine.py:159–164`): `n_init` (number of random restarts) is passed as `n_iter` (number of EM iterations). `hmmlearn.GaussianHMM` has no `n_init` parameter; `n_iter=10` is far too few EM iterations (default 100). The spec wants multiple random restarts per candidate — this runs exactly one.

**`min_train_bars` has silent `*2` multiplier** (`hmm_engine.py:137`): `if len(features) < self.min_train_bars * 2`. With default 252, the actual requirement is 504, which matches the spec numerically but callers passing `min_train_bars=504` would get a 1008-bar requirement.

**`get_regime_flicker_rate()` returns a count, not a rate** (`hmm_engine.py:483–492`): name and spec contract say "rate" (changes per window) but the return value is a raw count.

**`get_strategy_for_regime` recomputes sort every call** (`regime_strategies.py:313–322`): O(n log n) per inference bar. Should be cached after each `update_regime_infos` call.

**`generate_signal(capital=1.0)` hard-codes capital** (`regime_strategies.py:97, 168, 241`): strategies compute dollar-sized positions from a hard-coded `capital=1.0`. The spec wants strategies to emit `position_size_pct ∈ {0.60, 0.95}`; actual position sizing belongs in `risk_manager.py`.

**`save()` / `load()` lose fields** (`hmm_engine.py:517–564`): `n_regimes`, `best_bic`, and several `RegimeInfo` fields (`recommended_strategy_type`, `max_leverage_allowed`, etc.) are not round-tripped.

**Silent `except Exception` swallows convergence failures** (`hmm_engine.py:183–185`): should log at ERROR level with `exc_info=True`.

**`RegimeState.timestamp` uses wall-clock time** (`hmm_engine.py:478`): in backtesting this should be the bar timestamp, not `datetime.now()`.

**Transition confirmation never reduces size by 25%** (`hmm_engine.py` + `regime_strategies.py`): spec §5 "during transition: keep previous regime, reduce sizes by 25%". `is_confirmed` is tracked but never checked by the strategy layer.

## Positive Observations

1. **Forward algorithm is correct.** `predict_filtered` works in log-space with `logsumexp`, seeds from `startprob_`, and caches `_previous_proba`. This is the most critical part of the spec.

2. **Explicit Viterbi ban is documented.** Module docstring calls out why `model.predict()` is prohibited. Good institutional memory.

3. **Return-vs-volatility sort separation is correct.** `_label_regimes` sorts by return; `get_strategy_for_regime` sorts by volatility — exactly the invariant emphasized in both spec documents.

4. **BIC formula and parameter counting are correct.** Per-covariance-type parameter counts are right.

5. **ADX and ATR computed from first principles.** Easier to audit than a `ta` library call.

6. **Dataclasses are well-typed and clearly structured.** `RegimeInfo`, `RegimeState`, and `StrategySignal` fields are readable.

7. **Flicker detection logic is present.** Count-based sliding window, just needs a naming fix.
