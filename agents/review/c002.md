### CRITICAL-4 — Forward algorithm applies `transmat_` in the wrong direction

`hmm_engine.py:407–413`

`hmmlearn` stores `transmat_[i, j] = P(state_t = j | state_{t-1} = i)`. The correct forward step is:

```
alpha_t(j) = sum_i  alpha_{t-1}(i) * T[i, j]
```

In log-space that is:

```python
# correct
log_alpha_t[j] = logsumexp_i( log_alpha_{t-1}[i] + log(T[i, j]) )
```

The code transposes first, then broadcasts on the wrong axis:

```python
log_transposed = np.log(model.transmat_.T + 1e-300)   # T^T[a,b] = T[b,a]
log_alpha_transitioned = logsumexp(
    log_alpha_prev[:, np.newaxis] + log_transposed,    # (i,1) + (a,b) -> sums over axis=0
    axis=0
)
```

`log_alpha_prev[:, np.newaxis] + log_transposed` is `(a,b)` = `log P(prev=a) + log T[b,a]`.  
`logsumexp(..., axis=0)` sums out `a`, yielding `log sum_a P(a) * T[b,a]` — that is `(T @ prev)[b]`, **not** `(prev @ T)[b]`.

Empirical check with an asymmetric transition matrix:

```
true forward  (prev @ T): [0.590  0.250  0.160]
code result   (T @ prev): [0.590  0.240  0.170]
```

The two match only when `T` is symmetric. The "sticky BULL, fast-transition-to-CRASH" structure the bot is designed to detect is maximally asymmetric.

**Impact:** every regime posterior after bar 0 is wrong. `TestNoLookAheadBias` doesn't catch it because both the "short" and "long" runs use the same wrong direction — they agree on the wrong answer.

**Fix:** drop the transpose:

```python
log_trans = np.log(model.transmat_ + 1e-300)   # shape (n, n): T[i,j] = P(i→j)
log_alpha_transitioned = logsumexp(
    log_alpha_prev[:, np.newaxis] + log_trans,  # sum over previous state i on axis 0
    axis=0
)
```

Add a regression test with a known asymmetric `transmat_`, compare `predict_filtered` to `hmmlearn`'s own `predict_proba` on the same prefix.

### CRITICAL-5 — `StrategyOrchestrator` cache is never populated on first use

`regime_strategies.py:394–407`

```python
if regime_infos != self._last_regime_infos and len(regime_infos) == len(
    self._last_regime_infos
):
    self.update_regime_infos(regime_infos)
```

On the very first call `self._last_regime_infos == {}` (empty), so `len({}) != len(fresh_infos)` and the condition is `False`. `update_regime_infos` is never called. `_cached_strategy_map` stays empty. Every lookup falls through to the `uncertainty_strategy` fallback — the most defensive strategy (60% allocation, 1.0x leverage) — silently, forever.

```
empty cache result:   len=0  → uncertainty_strategy every call
expected cache result: {0: LowVolBull, 1: MidVolCautious, 2: HighVolDefensive}
```

**Impact:** the orchestrator's purpose is entirely defeated on the happy path. Trades are sized as if the model is always uncertain.

**Fix:**

```python
def get_strategy_for_regime(self, regime_id, regime_infos):
    if regime_infos and regime_infos != self._last_regime_infos:
        self.update_regime_infos(regime_infos)
    strategy = self._cached_strategy_map.get(regime_id)
    if strategy is None:
        logger.warning("Unknown regime_id=%s; using uncertainty_strategy", regime_id)
        return self.uncertainty_strategy
    return strategy
```

Also remove the dead code at lines 408–436, which is unreachable after `return self.uncertainty_strategy`.

### CRITICAL-6 — `_label_regimes` is non-deterministic when two regimes share equal expected return

`hmm_engine.py:260`

```python
sorted_ids = np.argsort(regime_returns)   # unstable sort
```

`np.argsort` is not guaranteed stable. When two regimes have identical (or near-identical) mean returns — which routinely happens when the BIC-optimal k is smaller than `n_candidates` and Baum-Welch merges two states — the label assignment can swap between runs:

```
run 1: regime 2 → NEUTRAL, regime 1 → BEAR
run 2: regime 1 → NEUTRAL, regime 2 → BEAR
```

Downstream effects:
- `RegimeState.label` flips between "NEUTRAL" and "BEAR" on re-training from identical data.
- Strategy routing (via `LABEL_TO_STRATEGY`) changes.
- Historical logs become non-joinable across retrains.

**Fix:** stable sort with an explicit tiebreaker:

```python
sorted_ids = np.lexsort((
    np.arange(len(regime_returns)),  # tertiary: regime_id (stable)
    variances,                        # secondary: volatility asc
    regime_returns,                   # primary: return asc
))
```

(`np.lexsort` is always stable.)

### CRITICAL-7 — `HMMEngine` inference state is not thread-safe

`hmm_engine.py:115–120, 402–447`

`predict_filtered` mutates six instance attributes on every call:

```python
self._previous_proba = posterior
self._current_regime = current_regime
self._consecutive_bars = ...
self._regime_history.append(current_regime)
self._regime_history.pop(0)
self._is_flickering = ...
```

No lock exists. A monitoring thread calling `predict_filtered` while the trading loop is also calling it will corrupt `_previous_proba`, causing the next bar's forward step to consume the wrong prior. The error is silent: a valid-looking float posterior emerges.

**Fix (preferred):** separate trained model from inference state:

```python
class RegimeFilter:
    """One per live stream (symbol × timeframe). Wraps a shared HMMEngine."""
    def __init__(self, engine: HMMEngine):
        self.engine = engine
        self._previous_proba: Optional[np.ndarray] = None
        self._current_regime: Optional[int] = None
        self._consecutive_bars: int = 0
        self._regime_history: list[int] = []
        self._is_flickering: bool = False

    def step(self, feature_vector: np.ndarray, *, timestamp=None) -> RegimeState:
        ...
```

At minimum, add a `threading.RLock` around the mutable block and document concurrent access restrictions.

### HIGH-11 — `predict_filtered` has no input validation — NaN/shape mismatches fail silently

`hmm_engine.py:348–396`

Three failure modes that produce wrong-but-plausible output:

1. **NaN in `feature_vector`**: NaN flows through `diff`, through `solve`, gives NaN Mahalanobis, NaN emission, NaN posterior. `argmax(NaN_array)` returns 0. The engine assigns `_current_regime = 0` with probability NaN and continues. No exception.

2. **Wrong feature count**: `diff = obs - model.means_[i]` broadcasts incorrectly. `solve` raises `ValueError: incompatible dimensions` — but only `np.linalg.LinAlgError` is caught. The uncaught `ValueError` crashes the trading loop.

3. **Stale `_previous_proba` after retrain** (different `n_components`): `log_alpha_prev` is shape `(3,)` from the old model, new model has `transmat_` shape `(4,4)`. Broadcasting produces a `(3,4)` matrix; `logsumexp(..., axis=0)` gives shape `(4,)` — appears to work, is numerically garbage.

**Fix:**

```python
obs = np.atleast_2d(np.asarray(feature_vector, dtype=np.float64))
if obs.shape != (1, self.model.n_features):
    raise ValueError(f"Expected shape (1, {self.model.n_features}); got {obs.shape}")
if not np.all(np.isfinite(obs)):
    raise ValueError("predict_filtered received NaN/Inf in feature_vector")
if self._previous_proba is not None and len(self._previous_proba) != self.model.n_components:
    logger.warning("Stale inference state after retrain; resetting")
    self.reset_inference_state()
```

Also broaden the `except` on line 372 to `except (np.linalg.LinAlgError, ValueError)`.

### HIGH-12 — Emission probability crashes on singular covariance (scipy vs numpy `LinAlgError` mismatch)

`hmm_engine.py:362–374`

```python
from scipy.linalg import solve
try:
    cov_inv_diff = solve(cov, diff.T, assume_a="pos")
    ...
except np.linalg.LinAlgError:
    log_emissions[i] = -np.inf
```

`scipy.linalg.solve` raises `scipy.linalg.LinAlgError` (distinct from `np.linalg.LinAlgError` on some scipy versions). The `except` clause does not catch it, so a singular covariance crashes `predict_filtered` entirely instead of setting that state's emission to `-inf`.

Additional issues:

- `log_det = np.log(np.linalg.det(cov))`: for feature dimension k > ~10, `det` underflows to 0 and `log(0) = -inf` — no exception, but emission becomes `-inf` silently for every bar once the regime's covariance gets large. Use `np.linalg.slogdet`:

```python
sign, logdet = np.linalg.slogdet(cov)
if sign <= 0:
    log_emissions[i] = -np.inf
    continue
log_det = logdet
```

- The `tied` covariance branch (lines 375–381) has **no try/except** at all. A singular tied covariance crashes uncaught.

- `from scipy.linalg import solve` is re-imported inside the state loop on every bar. Move to module top.

### HIGH-13 — ADX ±DM computation is mutually contaminated

`feature_engineering.py:107–127`

```python
plus_dm = pd.Series(
    np.where((plus_dm > minus_dm) & (plus_dm > 0), plus_dm, 0), ...
)
minus_dm = pd.Series(
    np.where((minus_dm > plus_dm) & (minus_dm > 0), minus_dm, 0), ...
)
```

`minus_dm` on line 2 reads the **already-mutated** `plus_dm` (which has been zeroed wherever `minus_dm` previously won). As a result:

- On a bar where raw `minus_dm` beat raw `plus_dm`, `plus_dm` was set to 0. Now the check `minus_dm > plus_dm` becomes `minus_dm > 0`, which is a weaker condition — **both** ±DM can be non-zero on the same bar, violating the ADX definition.

**Fix:** compute from the original raw values before either is mutated:

```python
raw_plus  = high - high.shift(1)
raw_minus = low.shift(1) - low

plus_dm_arr  = np.where((raw_plus  > raw_minus) & (raw_plus  > 0), raw_plus,  0.0)
minus_dm_arr = np.where((raw_minus > raw_plus)  & (raw_minus > 0), raw_minus, 0.0)

plus_dm  = pd.Series(plus_dm_arr,  index=high.index)
minus_dm = pd.Series(minus_dm_arr, index=high.index)
```

### HIGH-14 — BIC model selection accepts non-converged models

`hmm_engine.py:163–190`

`hmm.fit()` can silently terminate before convergence (when `n_iter` is exhausted). The code never checks `hmm.monitor_.converged`. A non-converged model's log-likelihood is lower than the true maximum, so its BIC is artificially inflated — it will appear worse than converged models and be correctly penalised. However, if **all** candidates fail to converge, the "best" model selected is whichever reached the highest partial likelihood in fewer iterations, which is a poor proxy for model fit.

**Fix:** log convergence status for every candidate and skip non-converged models when at least one converged:

```python
if not hmm.monitor_.converged:
    logger.warning(
        "HMM n_components=%d did not converge in %d iterations",
        n_components, self.n_init
    )
    # Don't skip — use as fallback — but mark it
    converged_models.append((bic, hmm, False))
else:
    converged_models.append((bic, hmm, True))
```

Then prefer converged ones; fall back to best non-converged only if all failed.

### HIGH-15 — `reset_inference_state()` is never called after `train()` — stale prior leaks into next inference stream

`hmm_engine.py` (end of `train`)

After `train()` completes, `_previous_proba` still holds the posterior from the most recent `predict_filtered` call against the **old** model. The next call to `predict_filtered` will use this stale posterior as the prior for the new model's forward step — wrong regime count, wrong probability values.

**Fix:** add at the end of `train()`:

```python
self.reset_inference_state()
logger.info("Inference state reset after retrain")
```

### MEDIUM — `save()` / `load()` does not persist hyperparameters

`hmm_engine.py:520–572`

`__init__` parameters (`n_candidates`, `n_init`, `covariance_type`, `min_train_bars`, `stability_bars`, `flicker_window`, `flicker_threshold`, `min_confidence`) are not included in the pickle payload. On reload, the engine is reconstructed with defaults. A model trained with `min_confidence=0.70` silently reverts to 0.55 after reload, changing `RegimeState.is_uncertain` semantics.

**Fix:** serialize all hyperparameters alongside the model weights, and assert on load that the restored hyperparameters match what the caller expects.

### MEDIUM — `_previous_proba + 1e-300` log floor corrupts peaked posteriors over many bars

`hmm_engine.py:404, 407, 416`

With a peaked posterior `[1.0, 0.0, 0.0]`, `log(0 + 1e-300) ≈ -690.7`. Over 100 bars of a 5-state HMM, the non-active states accumulate `≈ -690` in `log_alpha_prev`. Combined with `log(det(cov))` for full-covariance high-dimensional features, `log_alpha_unnorm` entries can push outside the dynamic range of `float64 exp`, producing `nan` after `logsumexp`.

**Fix:** after computing `log_normalizer`, guard:

```python
log_normalizer = logsumexp(log_alpha_unnorm)
if not np.isfinite(log_normalizer):
    logger.warning("Numerical underflow in forward step; rebooting to uniform prior")
    log_alpha_unnorm = np.full_like(log_alpha_unnorm, -np.log(self.model.n_components))
    log_normalizer = logsumexp(log_alpha_unnorm)
```

### MEDIUM — `uncertainty_strategy` is overloaded as both "risk-off" and "bug sentinel"

`regime_strategies.py:334, 404–406`

Two distinct failure modes both return `uncertainty_strategy`:

1. Regime probability below threshold (legitimate, risk-off)
2. `regime_id` not found in strategy cache (bug, should alert)

These are indistinguishable in the trade log. Mode 2 is caused by the cache-population bug (CRITICAL-5) — so what looks like "cautious trading" is actually "trading with a broken cache".

**Fix:** route case 2 to a zero-size "safe-mode" fallback with `action="exit"` and log at ERROR level, distinct from the legitimate uncertainty path.

- The `logsumexp` normalisation step in `predict_filtered` is structurally correct — the algorithm is right in form, only the transmat direction is wrong.
- The per-regime emission calculation explicitly handles all four `covariance_type` variants rather than relying on `hmmlearn` internals — defensively correct against future API changes.
